# Platform Adapter 추가 가이드

## 목표
신규 플랫폼을 추가할 때 도메인 로직 변경을 최소화하고, adapter 계층에 변경을 국소화합니다.

## 절차
1. **요구사항 정의**
   - 조회/답글 기능 범위, 제한사항, 호출 빈도 정리.
2. **Contract 확정**
   - 내부 표준 타입(리뷰 조회/답글 등록 요청·응답) 우선 설계.
3. **인증 전략 선택**
   - 세션 쿠키/토큰 갱신 방식, 만료 처리 정책 정의.
4. **Adapter 스켈레톤 구현**
   - `fetchReviews`, `createReply` 등 필수 메서드 구현.
5. **에러 매핑 구현**
   - 플랫폼 에러를 재시도 가능/불가능으로 분류.
6. **관측성 연결**
   - 로그 필드(플랫폼, 계정, 요청ID, 결과코드) 표준화.
7. **통합 테스트/샌드박스 검증**
   - placeholder 환경 기반 계약 테스트 수행.
8. **점진적 릴리스**
   - 일부 계정/스토어 대상 canary 실행 후 전체 확장.

## 체크리스트
- [ ] 도메인 레이어에 플랫폼 고유 문자열이 유입되지 않았다.
- [ ] 내부 Contract를 기준으로 요청/응답이 변환된다.
- [ ] 인증 실패/권한 부족/속도 제한 처리 경로가 있다.
- [ ] 재시도(backoff) 정책과 비재시도 조건이 명확하다.
- [ ] 실패 시 감사 로그와 알림이 남는다.
- [ ] 민감정보 마스킹 규칙이 적용된다.
- [ ] 운영 토글(활성/비활성, dry-run)이 준비되어 있다.
